# D 模块开发总结与报告材料：语义分析

## 1. 模块目标

语法分析只能证明输入符合 ToyC 文法，不能判断标识符、作用域、函数签名和控制流是否合法。D 模块在 Parser 生成 AST 后执行静态语义检查，并保证只有合法 AST 才进入 IR 与 RISC-V 后端。

完整位置：

```text
Lexer -> Parser -> AST -> SemanticChecker -> IR -> RISC-V32
```

## 2. 总体设计

语义分析采用一次按源码顺序遍历 CompUnit 的方式，并维护三类状态：

1. **作用域栈**：每层使用哈希表记录变量、常量、参数或函数；
2. **函数上下文**：记录当前函数名称与返回类型；
3. **循环深度**：判断 `break`、`continue` 是否位于 `while` 内。

函数调用必须出现在被调函数定义之后，因此不能在开始时预先收集所有函数。遍历到一个函数时，先登记当前函数签名以支持递归，再检查函数体。后续函数在此时仍不可见。

## 3. 符号表

符号分为：

```text
Variable
Constant
Parameter
Function
```

变量、常量、参数和函数使用同一普通标识符命名空间。同层重名报错，向外层查找实现嵌套作用域和屏蔽规则。声明的初始化式先检查，之后才把名字插入符号表，从而落实“使用必须发生在声明之后”，并避免变量在自身初始化式中错误引用自己。

常量符号额外保存编译期 `int32` 值，供后续常量声明、短路判断、除零检查和控制流分析使用。

## 4. 表达式类型检查

ToyC 的普通值类型只有 `int`，函数返回类型为 `int` 或 `void`。表达式遍历返回 `ValueType`：

- 数字、变量、常量、参数、一元运算、二元运算返回 `int`；
- 函数调用返回函数签名声明的类型；
- `void` 调用仅允许单独作为表达式语句；
- 条件、初始化式、赋值右值、函数实参和 `int` return 都要求 `int`。

函数调用还检查可见性、被调用标识符是否确实为函数、实参数量和逐项类型。

## 5. 常量表达式

常量初始化式分两步处理：

1. 遍历完整表达式，确保只含数字、此前声明的常量和合法运算；
2. 按 32 位有符号整数规则求值。

求值支持：

```text
+ - * / %
< > <= >= == !=
&& || !
一元 + -
```

`&&` 与 `||` 按短路规则求值。例如：

```c
const int A = 0 && (1 / 0);
```

右侧不会执行，因此不会产生除零错误；但右侧仍会接受符号和类型检查。实现还检测可确定的除零、取模零与有符号常量溢出。

## 6. 控制流和 return

每条语句返回控制流摘要，记录是否可能：

```text
正常落到下一条语句
return
break
continue
无限循环
```

Block 按顺序组合摘要；if 合并两条分支；while 消化循环体的 break/continue。若条件是常量，控制流分析会排除不可能分支。

对 `int` 函数，语义分析要求普通控制流不能到达函数末尾。无穷循环不会落到函数末尾，因此与 C 风格控制流检查一致。

## 7. 测试设计

`tests/semantic_checker_tests.cpp` 直接构造 AST，以隔离 Lexer/Parser 尚未完成或联调造成的影响。测试覆盖：

- 合法综合程序；
- main 缺失、类型或参数错误；
- 重复声明和嵌套屏蔽；
- 使用未声明变量；
- 常量赋值、非法常量初始化；
- 短路求值和除零；
- 函数调用顺序、递归、参数数量；
- void 调用上下文；
- return 类型和分支完整性；
- break/continue 循环上下文；
- 参数重名与全局名字冲突。

## 8. 与其他模块的接口

输入：

```cpp
toycc::ASTNodePtr
```

调用：

```cpp
toycc::SemanticChecker checker;
checker.check(ast);
```

输出：检查成功后原 AST 继续交给 IR；失败时抛出 `toycc::SemanticError`。该设计与现有 Parser 抛异常的错误处理方式一致，主流程可在最外层统一捕获并输出到 `stderr`。

## 9. 当前限制与后续改进

现有 AST 节点不保存 Token 或源码位置，因此 SemanticError 无法报告精确行列。后续可以给 AST 基类增加 SourceLocation，或在关键节点保存对应 Token。若 IR 需要直接复用符号信息，也可以在保持 `check` 接口的基础上增加只读 SemanticResult，但当前任务所需检查已全部由 SemanticChecker 完成。
